BCPD Operating State v2.1 — Architecture

Audience: Internal Flagship / BCPD team plus technical reviewers. Generated: 2026-05-01. Source: docs/bcpd_operating_state_architecture.md + docs/bcpd_operating_state_architecture.mmd.

Plain English summary. We take raw extracts from DataRails, QuickBooks, ClickUp, the inventory closing report, the collateral reports, and the allocation workbooks; normalize them; reconcile their different vocabularies into a small set of canonical business entities (LegalEntity, Project, Phase, Lot, FinancialTransaction, etc.); and publish a single versioned JSON — operating_state_v2_1_bcpd.json — that an agent or human can query with confidence labels, caveats, and refusals built in. The system is read-only at the agent layer. It does not write back to source systems. It does not invent numbers. When a question can't be answered safely, the agent layer refuses rather than fabricating. The system is BCPD-only; org-wide rollups are explicitly out of scope.

Layer-by-layer

  1. Source files and systems — the original extracts as they arrive from upstream tools. We do not edit these. They live under data/raw/.
  2. Staging and normalization — Python pipelines that read the raw files and produce stable parquet/CSV under data/staged/. Two pieces of logic deserve callouts:
    • DataRails dedup: DR 38-col carries every posting line 2-3 times consecutively (2.16× multiplicity). Naive sum is wrong by ~2×; the build pipeline deduplicates on a 9-field canonical key. Raw staged_gl_transactions_v2.parquet is preserved unchanged.
    • VF lot-code decoder v1: VF's lot field encodes phase + lot in a single string for some projects (Harmony VF lot 1034 ≠ inventory Harmony lot 1034). The decoder routes lot strings to (canonical_phase, canonical_lot_number). All decoder rules ship confidence='inferred'.
  3. Crosswalks — translation tables that map source values to canonical names. PWFS2, Park Way, PARKWAY, and Parkway Fields all refer to one community. The crosswalk is the single place that resolution lives. Every row carries a confidence column.
  4. Canonical business entities — a small graph (LegalEntity, Project, Phase, Lot, FinancialTransaction, Account, CostCategory, InventorySnapshot, CollateralSnapshot, TaskState, Allocation). Every canonical row carries source_confidence (worst-link of contributing fields).
  5. Operating State v2.1 — operating_state_v2_1_bcpd.json is the canonical state document; three companion markdown files document field quality, worked queries, and the v2.0→v2.1 deltas.
  6. Agent / read-only Q&A layer — agent_context_v2_1_bcpd.md grounds the agent; bcpd_state_qa is a deterministic harness that answers 15 fixed questions with 10 enforced guardrails. Every answer carries direct_answer, evidence, confidence, caveats, source_files_used, and a cannot_conclude flag for refusals.

Why project + phase + lot matters

In some BCPD projects, the same lot number appears in multiple phases. Joining cost rollups on (project, lot) alone collapses different physical lots onto one inventory row.

The clearest example is Harmony: lot numbers 101–116 exist in two distinct phases — MF1 (multi-family / townhomes) and B1 (single-family). They are two different physical assets that happen to share a lot number. v2.0 used a flat (project, lot) join. v2.1 enforces the 3-tuple (project, phase, lot). The fix prevents a known ~$6.75M double-count risk on Harmony alone.

The 3-tuple discipline is enforced at three points in v2.1:

  • The VF lot-code decoder emits (phase, lot_number) per row.
  • Every lot in the JSON carries vf_actual_cost_3tuple_usd computed at the 3-tuple, plus vf_actual_cost_join_key as a constant string declaring the contract.
  • The agent context lists "Harmony joins require project + phase + lot" as Rule 4, and the Q&A harness has a guardrail (harmony_3tuple_required) that fires whenever a question asks about Harmony cost.

This is not a Harmony-only rule — it is the project-wide policy in v2.1. Harmony is where the discipline is visible; in other projects, lot-number collisions don't currently occur but the same key is used so future surprises are avoided.

Guardrails

These are enforced at the agent layer and assumed by every downstream consumer. They are constraints on what the system will and will not say.

#RuleMeaning
1BCPD-onlyCovers BCPD + BCPBL + ASD + BCPI. Does NOT cover Hillcrest, Flagship Belmont, Lennar, or external customers.
2Missing cost is missing, not zeroA project or lot with no GL row has cost=unknown/null, never $0.
3Org-wide v2 is blockedHillcrest and Flagship Belmont have GL only through 2017-02. Refused at the agent layer.
4Range / shell rows are NOT lot-level cost$45.75M lives at project+phase grain via vf_unattributed_shell_dollars. Do not allocate to specific lots without source-owner sign-off.
5Harmony joins require project + phase + lotFlat (project, lot) collapses MF1 and B1 lot 101 onto one inventory row.
6Inferred decoder rules are not source-owner-validatedEvery per-lot VF cost via the v1 decoder ships confidence='inferred'.
7DataRails raw sums are unsafe without dedupDR 38-col is 2.16× row-multiplied at source.
8QB Register is tie-out onlyDifferent chart of accounts (zero overlap with VF/DR). Use only for 2025 BCPD vendor / cash queries.
9Vertical Financials is one-sidedAsset-side debits only (3 account codes). Canonical lot-level cost basis; not a balanced trial balance.
10Read-only at the agent layerQ&A harness does not write back to source/staged/v1/v2.0/v2.1 files. Verified by tests/test_bcpd_state_qa_readonly.py.

The 15 fixed Q&A questions exercise these guardrails. Q7 (org-wide actuals) is the canonical refusal example: the harness returns cannot_conclude=True rather than fabricating a number.

What this is not

  • Not a dashboard. No HTML pages of charts ship with v2.1. The deliverable is a JSON document, several markdown contexts, and a deterministic Q&A harness.
  • Not a data warehouse. No SQL engine, no OLAP cube, no incremental ingestion. Pipelines are batch scripts.
  • Not org-wide. Out of scope by design. See guardrail 1.
  • Not a complete picture. 7 active BCPD projects + Lewis Estates have inventory but no GL — their cost is unknown. The 2017-03 → 2018-06 GL gap (15 months, zero rows) cannot be closed from the current dump. Structural gaps, not v2.1 defects.
  • Not source-owner-validated at the decoder level. Every decoder-derived per-lot cost is inferred.

What this is

  • A source-backed operating-state layer with explicit confidence, caveats, and source provenance for every value.
  • A safe substrate for AI agents and automation. The agent context's enforced rules + the harness's guardrails mean an LLM has no excuse to invent numbers, conflate phases, or report missing cost as zero.
  • A reproducible, deterministic pipeline. Re-running on the same inputs produces identical outputs.
  • An additive system. v0, v1, and v2.0 outputs are preserved unchanged. v2.1 sits alongside.
