# Tools — BCPD Operating State Skill

_How the Skill invokes the existing six runtime tools. **No tool code lives
in this directory.** This Skill is a wrapper around `core/tools/bcpd_workflows.py`._

---

## The six tools

All six live in `core/tools/bcpd_workflows.py`, subclass `core.tools.base.Tool`,
and register cleanly with `core.agent.registry.ToolRegistry`.

| Skill capability | Tool class | Anthropic tool name | CLI command |
|---|---|---|---|
| Generate Project Brief | `GenerateProjectBriefTool` | `generate_project_brief` | `python -m bedrock.workflows.cli project-brief --project "<name>"` |
| Review Margin Report Readiness | `ReviewMarginReportReadinessTool` | `review_margin_report_readiness` | `python -m bedrock.workflows.cli margin-readiness --scope bcpd` |
| Find False Precision Risks | `FindFalsePrecisionRisksTool` | `find_false_precision_risks` | `python -m bedrock.workflows.cli false-precision --scope bcpd` |
| Summarize Change Impact | `SummarizeChangeImpactTool` | `summarize_change_impact` | `python -m bedrock.workflows.cli change-impact` |
| Prepare Finance / Land Review | `PrepareFinanceLandReviewTool` | `prepare_finance_land_review` | `python -m bedrock.workflows.cli meeting-prep --scope bcpd` |
| Draft Owner Update | `DraftOwnerUpdateTool` | `draft_owner_update` | `python -m bedrock.workflows.cli owner-update --scope bcpd` |

Each Tool subclass implements `to_api_schema()` returning the Anthropic
`tool_use` API shape `{name, description, input_schema}`. The Skill manifest
just lists these schemas — no new code.

## How a Skill wrapper invokes them

```python
# Pseudo-code for the Skill runtime — the exact wrapper depends on the
# Skill platform's tool-use shape.

from core.agent.registry import ToolRegistry
from core.tools.bcpd_workflows import (
    BcpdContext,
    register_bcpd_workflow_tools,
)

# 1. Build context once (loads v2.1 state from bundled paths).
ctx = BcpdContext()

# 2. Register all six tools.
registry = register_bcpd_workflow_tools(ToolRegistry(), context=ctx)

# 3. Schemas for the Anthropic tools= parameter.
tools = registry.to_api_schemas()

# 4. On a tool_use call from Claude, dispatch:
result_md = registry.dispatch(name=tool_name, inputs=tool_inputs)
```

That's the entire wrapper. No new Tool subclasses. No re-implementation.

## Why no duplication

Every line of tool code lives in `core/tools/bcpd_workflows.py`. Duplicating
it inside the Skill would:

- Double the surface area for bugs.
- Break the regression bar: the 31 tests in `tests/test_bcpd_workflows.py`
  cover the canonical implementations only.
- Risk drift between the CLI demos (regenerated by CI) and the Skill outputs.
- Violate the "no parallel workflow system" constraint that's been
  consistent across this project's hardening passes.

If the Skill platform requires tool definitions co-located in the Skill
package, prefer a thin re-export module (e.g., `skills/bcpd-operating-state/tools/__init__.py`
that does `from core.tools.bcpd_workflows import *`) over a copy. Today,
this directory has no Python code — the wrapper plumbing is enumerated above
as design intent only.

## Read-only contract

All six tools are read-only against v2.1 state. They do **not**:

- mutate `output/operating_state_v2_1_bcpd.json`
- mutate the 46 `output/agent_chunks_v2_bcpd/` files
- mutate any of the seven protected files listed in `state/README.md`
- write outside `output/runtime_demo/`
- call any connector / live source system

This is enforced by `tests/test_bcpd_workflows.py::test_workflow_tools_did_not_modify_protected_files`,
which takes a sha256 snapshot of all seven protected files, runs every
tool, and re-snapshots — assertion fails if any byte changes.

## Provenance + caveats

Each Tool inherits `_track()` and `_provenance_section()` from `core.tools.base.Tool`.
When `include_provenance=True` (the default), markdown outputs append a
`## Data Provenance` section listing which steps were deterministic
(certain) vs probabilistic (estimated).

For the BCPD workflow tools, the v1 VF decoder is the dominant
probabilistic step — every per-lot cost figure inherits its
`inferred (v1 decoder)` confidence label. Tools surface this in the
**Confidence boundaries** table that ships in three of the six tools
(`generate_project_brief`, `review_margin_report_readiness`,
`find_false_precision_risks`).
