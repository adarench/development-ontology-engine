# `state/bcp_dev/`

Versioned, hand-authored state for BCP Dev allocation and accounting v0.2. Read by tools in `core/tools/bcp_dev_workflows.py`.

## What lives here

| File | Purpose |
|---|---|
| `allocation_workbook_schema_v1.json` | Sheet-by-sheet, column-by-column schema of the Flagship Allocation Workbook v3 (5 sheets). Per-column source-system mapping and input / computed / output classification. |
| `allocation_input_requirements_v1.json` | Per-(community, phase) checklist of inputs required to compute each Per-Lot Output field. Drives `check_allocation_readiness`. |
| `per_lot_output_schema_v1.json` | Canonical shape of the Per-Lot Output table (Community, Phase, Job Id, Lot Type, Lots, Direct, Indirect, Land, Water, Warranty, Total Cost, per-lot derivatives, Sales Price, Margin). Each field carries a `compute_status` flag. |
| `source_crosswalks_v1.json` | Crosswalk tables (ClickUp subdivision → QBD customer; QBD customer → workbook community; phase identifier → LND job; SctLot variants → canonical; cost-type label → CostPool; etc.). |
| `accounting_event_triggers_v1.json` | Materialized triggers (from→to status changes that should fire each AccountingEvent), with required-field checks. Derived from `process_rules/clickup_gl_event_map_v1.json`. |
| `source_file_manifest_v1.json` | Every source file currently in play. `path`, `source_system`, `last_modified`, `schema_version`, `used_by_tools`, `freshness_caveat`. Drives staleness warnings. |
| `bcp_dev_operating_state_v0_2.json` | Aggregated state snapshot (per-community / per-phase rollup with input availability, crosswalk status, MDA Day check status). The v0.2 analog of `output/operating_state_v2_1_bcpd.json`. |

## What does NOT live here

- `output/operating_state_v2_1_bcpd.json` — the BCPD v2.1 derived state. Frozen and untouched by v0.2.
- Process rules — those live at `state/process_rules/`.
- Originals of `.docx` / `.pptx` / `.xlsm` source documents — those land at `data/raw/process_docs/`.

## Conventions

- **Schema version in filename.** Bump `_v2` on breaking changes; never edit a versioned file in place after publication.
- **Hand-authored, not derived.** These files are written by humans (or Claude agents under instruction) based on observed source data. They are not regenerated by a pipeline.
- **`confidence`** — every value-bearing row carries a confidence label (see `docs/bcp_dev_process_ontology_v1.md` § "Confidence Vocabulary").
- **`source`** — every file declares which upstream source(s) it was authored from.

## Ownership

- **Allocation workbook schemas (3 files)** — Agent B on Day 3 of the v0.2 plan.
- **Crosswalks + source manifest (2 files)** — Agent D on Day 4.
- **`accounting_event_triggers_v1.json`** — Agent C / E (derived from process rules).
- **`bcp_dev_operating_state_v0_2.json`** — Lead, generated from the above; refreshed as upstream files change.
